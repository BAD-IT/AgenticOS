This milestone brings the "Site Reliability Engineering" (SRE) aspect to the Agentic OS.

By implementing Diagnostics and the Strict Idle Mode, the system becomes self-maintaining. It will automatically categorize fatal errors to save you debugging time, and it will use its downtime to clean up resources and learn from past successes (Skill Generation) without wasting compute on unnecessary external polling.

Here is the exact Markdown text for the Milestone 7 GitHub Issue.

## Issue: Milestone 7 - Diagnostics, Triage & Strict Idle Mode

**Objective:**

Implement automated diagnostic routines to classify fatal task errors (Triage) and establish a background daemon (Strict Idle Mode) that handles garbage collection, VRAM flushing, and token-saving Experience Consolidation (Skill Generation) when the system is inactive.

**1. Root Cause Analysis & Triage Worker (`src/maintenance/diagnostics.py`)**

- Create a dedicated background worker that monitors the PostgreSQL `Error Queue`.
- **Logic:** When a task exhausts all micro-loop retries and macro-loop Overseer interventions, it is marked as fatal. The Triage Worker parses the execution logs and classifies the failure into one of three strict domains:
    - **Category A (Host/Network):** Local environment issues (e.g., Docker volume full, missing local file, no internet).
    - **Category B (Third-Party):** External API provider failures (e.g., HTTP 401 Unauthorized, 503 Service Unavailable from external webhooks).
    - **Category C (Agentic OS):** Internal reasoning collapse, infinite tool loops, or severe LLM hallucinations.
- **UI Integration:** Append the categorized tag to the task payload so the frontend can display a clean, human-readable error summary instead of a raw stack trace.

**2. Strict Idle Mode Daemon**

- Implement a background scheduler that activates *only* when the `Tasks Queue` and `Pending Queue` are completely empty.
- **Garbage Collection:** Instruct the daemon to safely purge orphaned temporary files from the Artifact Runner and clean up outdated intermediate artifacts in the `/workspace/outbox`.
- **VRAM Flush:** Implement a programmatic trigger to unload heavy LLM models from GPU/Memory, freeing up host machine resources (critical for 64GB hardware limits) until a new task arrives.

**3. Experience Consolidation (Dynamic Skill Generation)**

- As the final step of the Idle Mode, implement the automated learning mechanism.
- **Analysis:** Scan the database for recently completed tasks that required multiple iterations to succeed.
- **Abstraction:** Trigger a localized LLM call to summarize the successful sequence of tool calls and logic into a compact, abstract "Skill" document (JSON/Markdown).
- **Vector Storage:** Embed this Skill document and save it natively into the unified PostgreSQL database utilizing the `pgvector` extension. (This allows future workflow nodes to retrieve the Skill via RAG and bypass the trial-and-error phase).

**Acceptance Criteria:**

- **Test 1 (Triage):** Simulating a task failure due to an invalid external API key successfully routes the task to the Error Queue and tags it with "Category B".
- **Test 2 (Idle Activation):** The system successfully detects empty queues, triggers the Idle Mode daemon, and deletes a mock temporary file from the outbox.
- **Test 3 (Skill Generation):** Providing the daemon with a mock historical state of a complex, successfully solved task results in the generation of a condensed Skill embedding safely stored in the `pgvector` database table.
