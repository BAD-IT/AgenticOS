## Issue: Milestone 2 - Docker Topology, Persistence & Execution Sandbox

**Objective:**

Containerize the Agentic OS using a Zero-Trust architecture. Establish the unified PostgreSQL database, configure hot-reloading for rapid local development, and isolate the AI code execution environment into a sandboxed sibling container.

**1. Main Orchestrator & Persistence (`docker-compose.yml`)**

- Define the core `orchestrator` service running the Python environment.
- Configure the `postgres` service using an image that natively includes `pgvector` (e.g., `pgvector/pgvector:pg16`).
- Implement Docker bind mounts for local host integration:
    - Mount local `./workspace/inbox` to the container (Read-Only context).
    - Mount local `./workspace/outbox` to the container (Read-Write for generated files).
- Enable hot-reloading for the `orchestrator` by mapping the local `./src` directory directly into the container so file changes apply instantly without rebuilding.

**2. The Artifact Runner (Zero-Trust Sandbox)**

- Add a sibling container named `artifact_runner` to the `docker-compose.yml`.
- **Security Constraints:** Strip all database credentials, API keys, and internal network access from this container.
- **Resource Limits:** Enforce strict hardware caps (e.g., using Docker's `deploy.resources.limits` for CPU and Memory) to prevent runaway AI-generated loops from crashing the host.
- **Shared Storage:** Mount *only* the `./workspace/outbox` directory so the Orchestrator can write code files, and the Runner can execute/compile them safely.

**3. Database Initialization (`init.sql`)**

- Create an initialization script that automatically runs when the PostgreSQL container starts for the first time.
- Ensure the script explicitly activates the `pgvector` extension (`CREATE EXTENSION IF NOT EXISTS vector;`).
- Pre-create the table structures for the 7 system queues (User Input, Tasks, Error, Pending, Notification, Review, Result Output) mapping to the Pydantic schemas from Milestone 1.

**Acceptance Criteria:**

- Running `docker-compose up -d` successfully starts all three containers (Orchestrator, PostgreSQL, Artifact Runner) without errors.
- Modifying a Python file in the local `./src` directory instantly reflects inside the running Orchestrator container.
- A network ping test confirms the `artifact_runner` container cannot reach the `postgres` container's port.
- Files manually placed in the local `./workspace/outbox` appear inside both the Orchestrator and the Artifact Runner containers.
