This is the milestone where the OS gets its "hands" and learns to interact with the physical environment safely.

By connecting the cognitive engine to the Artifact Runner, we enable the system to compile code, execute scripts, and manipulate files without endangering your host machine. We also implement the targeted file patching here to ensure the agent doesn't waste thousands of tokens rewriting entire files.

## Issue: Milestone 4 - Zero-Trust Sandbox & Tools (Targeted Patching)

**Objective:**

Connect the Orchestrator's `Node_Tool_Execution` to the isolated `artifact_runner` container. Implement secure File I/O capabilities and token-efficient file patching tools to prevent full-file rewrites.

**1. Sandbox Execution Interface (`src/tools/sandbox_exec.py`)**

- Implement a communication layer (using the Docker SDK or subprocess `docker exec`) allowing the Orchestrator to trigger commands inside the `artifact_runner` container.
- **Error Handling:** Capture standard output (`stdout`) and standard error (`stderr`) accurately. If a script crashes (e.g., compilation error or runtime exception), pass the exact error trace back to the LangGraph `GraphState` so the LLM can learn and retry.
- **Timeout Constraints:** Enforce strict execution timeouts (e.g., 30 seconds). If a generated script enters an infinite loop inside the sandbox, the Orchestrator must forcefully kill the execution and log a failure.

**2. File I/O Tooling (`src/tools/file_io.py`)**

- Build deterministic tools for file system interaction restricted exclusively to the predefined bind mounts.
- **Read Tool:** Allow reading from `/workspace/inbox` and `/workspace/outbox`.
- **Write Tool:** Allow file creation/writing strictly limited to `/workspace/outbox`. Any attempt to write outside this directory must automatically trigger a system block.

**3. Targeted File Patching (Token Conservation)**

- Implement a `patch_file` tool to handle modifications to existing files.
- **Mechanic:** Instead of accepting a full file string, the tool must accept precise instructions (e.g., `line_number`, `search_string`, `replace_string`, or a unified `diff` format).
- **Backend:** Use standard tools (like `sed`, `awk`, or a Python AST parsing script) within the runner to apply the patch.
- **LLM Directive:** Update the `Node_Worker_Thinking` system prompt to strictly mandate the use of `patch_file` over `write_file` when modifying files larger than 50 lines.

**Acceptance Criteria:**

- **Test 1 (Execution):** The Orchestrator successfully compiles and runs a generated "Hello World" C++ program inside the `artifact_runner`, capturing the output.
- **Test 2 (Safety):** A generated Python script containing an infinite loop `while True: pass` is executed in the runner, safely killed by the Orchestrator after the timeout, and logs an error to the graph state without affecting the host.
- **Test 3 (Patching):** The LLM successfully modifies a specific function inside a 500-line Python file using the `patch_file` tool, leaving the rest of the file completely untouched.
